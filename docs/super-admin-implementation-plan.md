# خطة تنفيذ نظام السوبر أدمن الشامل

> **توجيه عام للموديل المنفذ:**
> - قاعدة البيانات: **Supabase** — لا تكتب SQL يدويًا في ملفات التطبيق؛ أي تغيير في المخطط (Schema) يتم عبر أدوات Supabase المتاحة (MCP / migrations) مع تفعيل RLS على أي جدول جديد.
> - لا تعيد بناء الموجود: المشروع فيه بنية أدوار جاهزة في `lib/admin/roles.ts` وحماية في `middleware.ts` وصفحات كثيرة جاهزة. مهمتك **الربط والتوسعة** وليس البناء من الصفر.
> - كل milestone مستقلة وقابلة للاختبار لوحدها. لا تنتقل للـ milestone التالية قبل التحقق من معايير القبول.
> - جميع واجهات الـ API الجديدة يجب أن تتحقق من أن المستخدم `super_admin` قبل التنفيذ (على مستوى الـ route نفسه وليس الـ middleware فقط).
> - اللغة: كل النصوص الظاهرة للمستخدم بالعربية مع دعم RTL.

---

## السياق الحالي (اقرأه قبل البدء)

- **الأدوار**: ثلاثة أدوار معرفة في `lib/admin/roles.ts`: `super_admin` / `maqraa_admin` / `academy_admin`، مع قائمة مسارات حصرية للسوبر أدمن (`SUPER_ADMIN_ONLY_PREFIXES`).
- **الحماية**: `middleware.ts` يمنع غير السوبر أدمن من مسارات الحوكمة ويعيد 403 للـ API.
- **آلية الوضع (Mode)**: يوجد `app/api/admin/mode/route.ts` (يدعم GET و PUT فقط) ومكونات `components/admin/admin-role-switcher.tsx` و `components/admin/admin-mode-banner.tsx` — لكن المكونات تستدعي `POST` وهو غير موجود (**باج حرج**).
- **الداشبورد**: `app/admin/page.tsx` ثابت وموجه للمقرأة فقط — لا يتغير حسب الوضع.
- **صفحات موجودة بالفعل** يجب الاستفادة منها: `audit-log`, `email-templates`, `settings` (تشمل maintenance + system + security), `users`, `contact-messages`, `activity-logs`, `security`, `backup`, `seo`, `homepage`, `theme`, `branding`, `role-management`, `analytics` (platform-overview).
- **قرار تصميمي متفق عليه**: التبديل بين الأوضاع يتم عبر **Segmented Control ظاهر في أعلى السايدبار** (وليس dropdown ولا tabs)، بثلاثة أزرار: النظرة الشاملة / المقرأة / الأكاديمية. الوضع الافتراضي عند الدخول: **النظرة الشاملة**.
- **ملاحظة**: الأدمن الحالي (الذي يدير المنصتين) سيتم إلغاؤه لاحقًا — السوبر أدمن هو من يحل محله.

---

## Milestone 0 — إصلاح الباج الحرج في تبديل الوضع

**الهدف:** جعل آلية تبديل الوضع تعمل فعليًا قبل البناء عليها.

**المهام:**
1. أضف معالج `POST` في `app/api/admin/mode/route.ts` (أو وحّد الكلاينت على `PUT` — اختر الأقل تغييرًا، والمفضل إضافة `POST` مع إبقاء `PUT`).
2. تأكد أن الـ route يتحقق من أن المستخدم `super_admin` قبل تغيير الوضع، وأن الوضع يُخزّن بشكل آمن (cookie httpOnly أو جلسة) وليس client-side فقط.
3. تحقق من أن `admin-role-switcher.tsx` و `admin-mode-banner.tsx` يعملان بعد الإصلاح (التبديل يغير السايدبار فعليًا).

**معايير القبول:**
- تبديل الوضع من المكونات لا يرجع 405.
- مستخدم غير سوبر أدمن يحاول استدعاء الـ API يحصل على 403.
- بعد التبديل، إعادة تحميل الصفحة تحافظ على الوضع المختار.

---

## Milestone 1 — مبدّل الوضع الجديد (Segmented Control في السايدبار)

**الهدف:** استبدال آلية التبديل الحالية بأزرار ظاهرة دائمًا أعلى السايدبار.

**المهام:**
1. أنشئ مكون `components/admin/mode-segmented-switcher.tsx`: ثلاثة أزرار ظاهرة (النظرة الشاملة / المقرأة / الأكاديمية)، الزر النشط مميز بخلفية ولون واضحين، بدون أي dropdown.
2. اعرضه في أعلى السايدبار داخل `components/dashboard-shell.tsx` **فقط عندما يكون المستخدم `super_admin`**.
3. عند التبديل: استدعاء API الوضع ثم إعادة توجيه إلى `/admin` (صفحة الداشبورد) مع تحديث السايدبار.
4. أبقِ `admin-mode-banner.tsx` كشريط تنبيه يظهر عند الوقوف داخل وضع منصة معينة، مع زر "الرجوع للنظرة الشاملة".
5. احذف/أوقف `admin-role-switcher.tsx` القديم بعد التأكد من عدم وجود استخدامات أخرى له.

**معايير القبول:**
- السوبر أدمن يرى الأزرار الثلاثة دائمًا في السايدبار؛ المدراء الآخرون لا يرونها إطلاقًا.
- التبديل يغيّر عناصر السايدبار فورًا حسب الوضع.
- الوضع الافتراضي عند أول دخول هو "النظرة الشاملة".

---

## Milestone 2 — داشبورد حسب الوضع (Mode-Aware Dashboard)

**الهدف:** `app/admin/page.tsx` يعرض داشبورد مختلف حسب الوضع الحالي.

**المهام:**
1. قسّم `app/admin/page.tsx` إلى ثلاثة مكونات داشبورد:
   - `dashboard-super.tsx`: نظرة شاملة تجمع أرقام المنصتين جنبًا إلى جنب (كروت المقرأة + كروت الأكاديمية + مؤشرات عامة للموقع). استفد من API الموجود `app/api/admin/platform-overview/route.ts` ووسّعه عند الحاجة.
   - `dashboard-maqraa.tsx`: انقل إليه محتوى الداشبورد الحالي (التلاوات، المقرئين... إلخ) كما هو.
   - `dashboard-academy.tsx`: داشبورد جديد بأرقام الأكاديمية (الدورات، الطلاب، المعلمين، الاشتراكات). ابحث أولًا عن جداول الأكاديمية الموجودة في Supabase عبر فحص الـ schema قبل كتابة أي استعلام.
2. الـ page الرئيسية تقرأ الوضع الحالي (server-side) وتعرض المكون المناسب.
3. للأدوار غير السوبر أدمن: `maqraa_admin` يرى داشبورد المقرأة، و`academy_admin` يرى داشبورد الأكاديمية — بدون مبدّل.
4. أنشئ/وسّع API endpoints للإحصائيات الناقصة (خاصة الأكاديمية) مع تحقق من الصلاحيات في كل endpoint.

**معايير القبول:**
- التبديل بين الأوضاع الثلاثة يعرض ثلاثة داشبوردات مختلفة فعليًا.
- داشبورد "النظرة الشاملة" يعرض بيانات حقيقية من المنصتين (وليس بيانات ثابتة).
- لا يوجد أي تسريب بيانات: كل endpoint يتحقق من الدور قبل الإرجاع.

---

## Milestone 3 — الإعدادات العامة للموقع (General Site Settings)

**الهدف:** صفحة إعدادات عامة على مستوى الموقع كله (وليس منصة بعينها) حصرية للسوبر أدمن.

**المهام:**
1. افحص أولًا صفحة `app/admin/settings/` الحالية ومكوناتها (`maintenance-settings`, `system-settings`...) — **قرر ما هو "عام للموقع" وما هو "خاص بالمقرأة"** وافصل بينهما بدلًا من التكرار.
2. أنشئ صفحة `app/admin/site-settings/` (سوبر أدمن فقط) تشمل: اسم الموقع، اللغة الافتراضية، المنطقة الزمنية، وضع الصيانة العام (للمنصتين)، روابط التواصل الاجتماعي.
3. خزّن الإعدادات في جدول إعدادات عام (key-value أو jsonb) في Supabase — افحص إن كان يوجد جدول settings حالي وأعد استخدامه إن أمكن.
4. أنشئ API route مع GET/PUT وتحقق صلاحيات، وأضف المسار الجديد إلى `SUPER_ADMIN_ONLY_PREFIXES` في `lib/admin/roles.ts`.
5. أضف الصفحة لقسم "الحوكمة والتحكم" في السايدبار.

**معايير القبول:**
- الحفظ والقراءة يعملان، والتغييرات تنعكس فعليًا (مثلًا وضع الصيانة يمنع الدخول للمنصتين).
- مدير المقرأة/الأكاديمية لا يستطيع الوصول للصفحة أو الـ API.

---

## Milestone 4 — صفحات المحتوى الثابت (Static Content Pages)

**الهدف:** تحكم السوبر أدمن في صفحات: من نحن، تواصل معنا، الشروط والأحكام، سياسة الخصوصية، الأسئلة الشائعة.

**المهام:**
1. أنشئ جدولًا لصفحات المحتوى في Supabase (slug، عنوان، محتوى، حالة نشر، تاريخ تحديث) مع RLS: قراءة عامة للمنشور، كتابة للسوبر أدمن فقط.
2. أنشئ صفحة إدارة `app/admin/content-pages/` (قائمة + محرر لكل صفحة) — استخدم نمط المحررات الموجود في `app/admin/homepage/_components/editors.tsx` كمرجع للأسلوب.
3. أنشئ API routes (list / get / update / publish) بتحقق صلاحيات.
4. أنشئ صفحات العرض العامة للزوار (`/about`, `/terms`, `/privacy`, `/faq`) تقرأ من الجدول — افحص أولًا إن كانت صفحات ثابتة موجودة بالفعل واستبدل محتواها الثابت بالمحتوى الديناميكي.
5. أضف المسار إلى `SUPER_ADMIN_ONLY_PREFIXES` والسايدبار.

**معايير القبول:**
- تعديل محتوى صفحة من لوحة السوبر أدمن يظهر فورًا في الصفحة العامة.
- الصفحات غير المنشورة لا تظهر للزوار.

---

## Milestone 5 — سجل التدقيق الشامل (Unified Audit Log)

**الهدف:** سجل موحد يغطي المنصتين: من فعل ماذا ومتى.

**المهام:**
1. **افحص الموجود أولًا**: يوجد `app/admin/audit-log/` و `app/admin/activity-logs/` و API لكل منهما — حدد الفروق بينهما وقرر التوحيد بدلًا من إنشاء ثالث.
2. تأكد أن جدول السجل يحتوي: المستخدم، الدور، الفعل، الكيان المتأثر، المنصة (maqraa/academy/site)، التوقيت، وبيانات إضافية (jsonb).
3. أنشئ helper موحد (مثل `lib/admin/audit.ts`) لتسجيل الأحداث، واستدعه من العمليات الحساسة: تغيير أدوار، تغيير إعدادات الموقع، تبديل الوضع، حذف مستخدمين، تعديل محتوى عام.
4. طوّر صفحة العرض: فلترة حسب المنصة والدور والفعل والفترة الزمنية، مع ترقيم صفحات (pagination) من السيرفر.
5. الصفحة الموحدة حصرية للسوبر أدمن.

**معايير القبول:**
- أي عملية حساسة (من القائمة أعلاه) تُسجّل تلقائيًا وتظهر في السجل.
- الفلاتر والترقيم يعملان من السيرفر (وليس تحميل كل السجل للكلاينت).

---

## Milestone 6 — إدارة المستخدمين على مستوى الموقع (Site-Wide User Management)

**الهدف:** السوبر أدمن يبحث ويدير أي مستخدم في أي منصة من مكان واحد.

**المهام:**
1. **افحص الموجود**: `app/admin/users/` موجودة مع صفحات أدوار وصلاحيات لكل مستخدم — حدد إن كانت تغطي مستخدمي الأكاديمية أيضًا أم المقرأة فقط.
2. وسّعها (أو أنشئ view موحد) لتشمل مستخدمي المنصتين مع عمود يوضح انتماء المستخدم (مقرأة / أكاديمية / كلاهما).
3. أضف إجراءات: تعطيل/تفعيل حساب، إعادة تعيين وصول، عرض نشاط المستخدم عبر المنصتين.
4. كل إجراء يُسجّل في سجل التدقيق (Milestone 5).
5. البحث والفلترة من السيرفر مع ترقيم صفحات.

**معايير القبول:**
- البحث عن مستخدم أكاديمية ومستخدم مقرأة من نفس الشاشة يعمل.
- تعطيل حساب يمنع دخوله فعليًا، ويظهر الإجراء في سجل التدقيق.

---

## Milestone 7 — قوالب الإشعارات والرسائل الموحدة

**الهدف:** تحكم السوبر أدمن في قوالب الإيميلات/الرسائل المشتركة بين المنصتين.

**المهام:**
1. **افحص الموجود**: `app/admin/email-templates/` و API الخاص بها موجودان — حدد ما إذا كانت مقصورة على منصة واحدة.
2. أضف حقل "النطاق" للقوالب (مقرأة / أكاديمية / عام) وفلترة حسبه.
3. القوالب "العامة" (ترحيب، استعادة كلمة سر، تحقق بريد) يتحكم فيها السوبر أدمن فقط؛ قوالب المنصة يتحكم فيها مدير المنصة.
4. أضف معاينة للقالب قبل الحفظ، وزر إرسال تجريبي (استفد من `app/api/admin/settings/test-email/route.ts` الموجود).

**معايير القبول:**
- مدير المقرأة يرى قوالب المقرأة فقط؛ السوبر أدمن يرى الكل.
- تعديل قالب عام ينعكس على الرسائل الفعلية المرسلة.

---

## Milestone 8 — صفحة التكاملات (Integrations)

**الهدف:** شاشة واحدة يرى منها السوبر أدمن حالة الخدمات الخارجية المتصلة.

**المهام:**
1. أنشئ صفحة `app/admin/integrations/` (سوبر أدمن فقط) تعرض حالة الاتصال بالخدمات: Supabase (قاعدة البيانات/التخزين)، خدمة البريد، أي خدمات فيديو أو دفع مستخدمة — **اكتشف الخدمات الفعلية من الكود** (مثل `video-settings` الموجودة) ولا تفترض.
2. لكل خدمة: حالة (متصلة/غير متصلة عبر فحص وجود env vars — **لا تعرض قيم المفاتيح أبدًا، فقط الحالة**)، وآخر فحص، وزر "اختبار الاتصال" حيث أمكن.
3. API route للفحص مع تحقق صلاحيات. لا يُخزّن أي مفتاح سري في قاعدة البيانات.

**معايير القبول:**
- الصفحة تعرض حالة صحيحة لكل خدمة دون كشف أي قيمة سرية.
- زر اختبار البريد يعمل (يرسل رسالة تجريبية).

---

## Milestone 9 — التنظيف والتوحيد النهائي

**الهدف:** إغلاق الفجوات وضمان اتساق النظام كله.

**المهام:**
1. راجع `SUPER_ADMIN_ONLY_PREFIXES` وتأكد أن كل المسارات الجديدة (site-settings, content-pages, integrations, والسجل الموحد) مضافة.
2. راجع السايدبار في `dashboard-shell.tsx`: قسم "الحوكمة والتحكم" يعرض كل الصفحات الجديدة، مرتبة في المجموعات الأربع: (هوية الموقع / الحوكمة والصلاحيات / إعدادات المنصة / الإشراف والمراقبة).
3. تأكد أن كل API جديد: يتحقق من الدور، يسجّل في audit log عند الكتابة، ويستخدم استعلامات parameterized.
4. راجع RLS على كل الجداول الجديدة في Supabase.
5. اختبار قبول نهائي: سيناريو كامل — سوبر أدمن يدخل، يرى النظرة الشاملة، يبدّل للمقرأة، يبدّل للأكاديمية، يرجع للشاملة، يعدّل إعدادًا عامًا، يتحقق من ظهوره في سجل التدقيق.

**معايير القبول:**
- مدير مقرأة أو أكاديمية لا يمكنه الوصول لأي صفحة أو API من صفحات الحوكمة (اختبار فعلي وليس نظري).
- السيناريو الكامل أعلاه يمر بنجاح من المتصفح.

## ترتيب التنفيذ والاعتماديات

```
M0 (إصلاح الباج) ← أساس لكل شيء
  └─ M1 (المبدّل الجديد)
       └─ M2 (الداشبوردات الثلاثة)
M3 (الإعدادات العامة)   ← مستقلة، يمكن بدؤها بعد M0
M4 (صفحات المحتوى)      ← مستقلة
M5 (سجل التدقيق)        ← يُفضّل قبل M6 و M7 (لأنهما يسجّلان فيه)
  ├─ M6 (إدارة المستخدمين)
  └─ M7 (قوالب الإشعارات)
M8 (التكاملات)          ← مستقلة
M9 (التنظيف النهائي)    ← الأخيرة دائمًا
```

**الأولوية المقترحة:** M0 → M1 → M2 → M3 → M5 → M6 → M4 → M7 → M8 → M9
